# 开放问题：可用 vllm-hust 实验数据验证的研究问题

本文档将综述中识别的关键挑战和研究缺口整理为可通过 vllm-hust/vllm-ascend-hust 实验验证的具体问题。每个问题映射到综述章节，给出待验证假设和所需实验基础设施。

目标：用 vllm-hust 的基准测试数据为综述的系统性判断提供实证支撑。

---

## 1. 国产后端执行路径分叉代价量化

**综述引用：** 第5节（硬件异构与后端碎片化）

**核心判断：** 共享路径分叉导致正确性边界不一致、维护成本持续上升。但目前缺乏量化证据说明分叉到底带来了多少实际性能损失和维护负担。

### 实验问题

- **Q1.1:** vllm-hust 在 Ascend 后端上 vs 上游 vLLM 在 NVIDIA GPU 上，同一模型（如 Qwen2.5-7B）的 feature parity 到底有多大差距？列出当前 Ascend 后端不支持或性能不达标的功能清单（graph mode、speculative decoding、prefix caching、PD 分离等）。
- **Q1.2:** 为了维持与上游主干的 merge-safe 关系，vllm-hust/vllm-ascend-hust 的插件化适配方式相比侵入式修改，在性能上付出了多少代价？对比同一功能的侵入式实现 vs 插件实现的吞吐差异。
- **Q1.3:** 上游 vLLM 每次大版本升级（如 v0.x → v0.y），vllm-ascend-hust 需要多少工程量来同步？统计历史上的 merge conflict 数量和解决时间。

### 所需基础设施
- vllm-hust + vllm-ascend-hust 在 Ascend 910B 上的部署
- 对照组：上游 vLLM 在 A100/H100 上的同模型部署
- Feature parity 跟踪表和 CI 回归数据

## 2. 复杂任务场景的状态治理压力

**综述引用：** 第5节（状态治理主路径：复杂任务场景的能力补齐）

**核心判断：** 多模态、结构化输出、工具调用等复杂请求进入运行时后，会同时冲击调度、缓存和图模式。关键问题不在接口名义上"可用"，而在端到端时延和稳定性。

### 实验问题

- **Q2.1:** 在 vllm-hust 上混合文本+多模态请求（如 Qwen2-VL），多模态请求的视觉 token 膨胀对共存的纯文本请求 p99 时延有多大干扰？量化资源争用效应。
- **Q2.2:** 结构化输出（grammar-constrained decoding）在 Ascend 后端上的额外开销有多大？对比开启/关闭 xGrammar 约束时的吞吐差异。
- **Q2.3:** 工具调用场景下的暂停-恢复（pause/resume）机制是否引入了不可忽略的 KV cache 管理开销？测量工具调用等待期间的 cache 驻留成本和恢复延迟。

### 所需基础设施
- vllm-hust 支持多模态（Qwen2-VL / InternVL2）的部署
- 混合负载生成器（文本 + 多模态 + 工具调用）
- 分阶段资源监控（prefill/decode/工具等待/恢复各阶段的 GPU 利用率和内存占用）

## 3. Prefix Caching 与 KV 状态治理在国产平台的有效性

**综述引用：** 第3节（状态治理主路径）、第5节（复杂任务场景）

**核心判断：** 前缀索引、多级驻留和状态迁移能力一旦不足，问题通常先在长上下文和高并发条件下显现。国产平台的 cache 命中率和状态迁移效率是否与主流 GPU 平台一致，缺乏公开数据。

### 实验问题

- **Q3.1:** 在 Ascend 910B 上，prefix caching 的实际命中率在不同工作负载分布下（ShareGPT对话、文档问答、代码补全）分别是多少？与 NVIDIA GPU 上的命中率对比。
- **Q3.2:** 长上下文场景（32K-128K tokens）下，KV cache 的分页管理在 Ascend 上是否引入了额外的碎片化开销？测量有效内存利用率。
- **Q3.3:** PD 分离（prefill-decode disaggregation）在 Ascend 双机/多机场景下，KV 传输延迟占端到端 TTFT 的比例是多少？网络带宽对收益的敏感度如何？

### 所需基础设施
- vllm-ascend-hust 开启 prefix caching 和 PD 分离
- 多种长度分布的工作负载生成器
- KV cache 命中率/驱逐率/碎片率监控
- 多机 HCCL/RoCE 网络环境

## 4. 压缩收益在国产平台的稳定性验证

**综述引用：** 第5节（压缩与成本主路径）

**核心判断：** 同一组量化参数在不同后端、不同上下文长度和不同负载强度下，可能分别表现为吞吐提升、尾时延恶化、状态迁移变多或精度明显波动。国产平台上这种不稳定性可能更加严重。

### 实验问题

- **Q4.1:** W4A16/W8A8 量化在 Ascend 910B 上的加速比，在不同 batch size 和 sequence length 下是否稳定？是否存在"量化反而变慢"的工作点？
- **Q4.2:** 量化后的 KV cache 压缩（如 KIVI 风格的 KV 量化）在 Ascend 上能否稳定减少内存占用同时不恶化 TPOT？精度损失的 perplexity 变化如何？
- **Q4.3:** 在实际 serving 场景下（而非离线 benchmark），低比特推理的单位 token 成本（tokens/秒/元）相比 FP16 到底优化了多少？把硬件功耗、内存占用和吞吐综合考虑。

### 所需基础设施
- vllm-ascend-hust 支持的量化路径（W8A8、W4A16）
- 功耗监控（npu-smi 或 ascend-dmi）
- 多维度 sweep：batch size × seq_len × quantization level
- 精度评估管线（wikitext perplexity、任务准确率）

## 5. 评测体系：从展示型 Benchmark 到决策型 Benchmark

**综述引用：** 第5节（评测体系与真实业务鸿沟）

**核心判断：** 当前评测主要围绕固定 batch、固定上下文长度的理想化场景展开，无法反映系统在生产环境中的真实表现。需要端到端工作负载评测。

### 实验问题

- **Q5.1:** 设计一套多阶段 disturbance benchmark（稳态→突发→缓存失效→负载漂移→恢复），在 vllm-hust 上跑出完整曲线。对比目前常用的单点 benchmark（如 benchmark_serving.py 的标准用法）结论是否一致。
- **Q5.2:** 多租户混部场景下（不同模型大小、不同 LoRA adapter），系统的吞吐稳定性（jitter）与 SLO 达成率是多少？对比单租户理想场景的结果差异有多大。
- **Q5.3:** 版本升级对性能的影响：vllm-hust 每次大版本更新后，同一 workload 的性能回归幅度是多少？是否有"修了新功能但旧场景变慢"的情况？

### 所需基础设施
- vllm-hust-benchmark 扩展多阶段评测脚本
- 多租户 serving 场景生成器
- 历史版本性能数据库（CI 回归数据）
- 真实业务工作负载回放能力

## 6. 上游演进与插件化架构的长期可持续性

**综述引用：** 第5节（上游演进与本地分叉维护）、第6节（能力契约与共享抽象）

**核心判断：** vllm-hust 代表了"开源主干复用增强"路线，其核心价值在于 merge-safe 的插件化适配方式。但这种方式的长期可持续性需要数据支撑。

### 实验问题

- **Q6.1:** vllm-ascend-hust 当前的代码变更中，有多少比例落在 plugin/registry/parser 等可独立演进的外围接口，多少比例不得不修改上游共享模块？统计分布作为"合并健康度"指标。
- **Q6.2:** 新模型接入（如新发布的 Qwen3、DeepSeek-V3）在 vllm-ascend-hust 上的适配周期是多少天？与侵入式修改路线（如 MindIE）的跟进速度对比。
- **Q6.3:** 在 vllm-hust 的 CI 回归中，上游更新导致的 Ascend 后端 break 频率是多少？修复平均耗时？

### 所需基础设施
- vllm-hust / vllm-ascend-hust 的 git 历史分析工具
- CI/CD 回归数据（break 率、修复时间）
- 新模型适配 timeline 记录

## 7. 控制面治理能力的实证验证

**综述引用：** 第6节（控制面、评测与交付的一体化治理）

**核心判断：** 控制面从运维脚本集合演化为连接评测、成本与交付的一体化治理层，是国产推理引擎从工程实现走向体系成熟的标志。

### 实验问题

- **Q7.1:** vllm-hust-dev-hub 的 quickstart + ascend-runtime-manager 自动化流程，能否将"从零到可服务"的时间从人工操作的 N 小时降低到 < 30 分钟？实测端到端部署时间。
- **Q7.2:** 版本矩阵稳定性：在 vllm-ascend-hust 的依赖链（CANN + torch_npu + vllm-hust + vllm-ascend-hust）中，有多少已知的版本组合是"可工作"的 vs "看似兼容但实际有问题"的？能否用 CI 自动验证？
- **Q7.3:** 灰度升级场景：在 Ascend 集群上从 vllm-ascend-hust v0.x 升级到 v0.y 时，能否做到零停机滚动升级？如果不能，实际的服务中断时间是多少？

### 所需基础设施
- vllm-hust-dev-hub 完整部署流程
- Ascend 多机集群环境
- 版本兼容矩阵自动化测试
- 升级回滚脚本和监控

## 优先级与综述章节映射

| 问题 | 难度 | 对综述的数据价值 | vllm-hust 所需能力 |
|------|------|-----------------|-------------------|
| Q1 (后端分叉代价) | 中 | 高 — 直接支撑§5第1个挑战 | feature parity 跟踪 |
| Q2 (复杂任务状态) | 高 | 高 — 直接支撑§5第2个挑战 | 多模态+工具调用支持 |
| Q3 (KV状态治理) | 中 | 高 — 支撑§3分析骨架 | prefix caching + PD分离 |
| Q4 (压缩稳定性) | 中 | 高 — 直接支撑§5第3个挑战 | 量化路径 + 成本监控 |
| Q5 (评测体系) | 低 | 极高 — 直接支撑§5第5个挑战 | benchmark 扩展 |
| Q6 (插件化可持续性) | 低 | 高 — 直接支撑§5第4个挑战 | git 分析 + CI 数据 |
| Q7 (控制面治理) | 中 | 中 — 支撑§6未来方向 | dev-hub 工具链 |

**建议执行顺序：** Q5 → Q6 → Q1 → Q3 → Q4 → Q2 → Q7

Q5（评测体系）是最高杠杆的第一步，因为它定义了所有后续实验应遵循的评测方法论，且难度最低（只需扩展已有 benchmark 脚本）。Q6（插件化可持续性）次之，因为它只需分析已有 git 历史和 CI 数据，无需新实验。

## 如何在综述中使用这些数据

每个实验结果应以如下方式集成到综述正文中：

1. 作为**挑战分析的实证段落**，放在第5节对应小节中，将"判断"升级为"判断+证据"
2. 作为综述第4节引擎比较的**量化补充**，使路线比较从定性走向定量
3. 作为第6节展望的**落地验证**，说明 vllm-hust 实践已经在朝综述指出的方向推进

目标不是把综述变成 systems paper，而是用可复现的实验数据增强综述论断的可信度，使其从"综合推理"上升到"综合推理+部分实证验证"。
