\section{关键挑战：主路径如何相互放大}

一个常见的场景可以说明问题的复杂性：团队好不容易让某款开源模型在国产后端上跑通了推理，吞吐指标也看得过去。但接下来做量化时发现，低比特算子在该后端上行为不一致，于是绕了一圈换方案；换完之后 KV Cache 管理逻辑也要跟着改，改了又触发了调度策略里一个隐式假设，导致尾时延飙升；最后想对比一下优化前后的效果，却发现团队用的 benchmark 根本没覆盖这种混合精度加长上下文的场景。这不是偶发故障，而是国产推理引擎面临的结构性困境：挑战之间会相互放大，一个路径上的适配缺口会沿着系统骨架向其他路径扩散。表\ref{tab:challenge-map} 把这种放大关系压缩为总览。

\subsection{执行与通信主路径：平台异构的持续扩散}

第 3 节已将后端能力契约列为执行路径的核心问题，这里要追问的是：为什么平台异构在国产推理引擎中不是"适配一次就完"的事？

答案在于硬件代际能力差异大且缺少统一适配层。vLLM-Ascend、vLLM-MLU 与 SGLang Ascend 的公开材料都体现出对上游主线版本、CANN SDK 和 torch\_npu 三方联合对齐的强依赖\citep{vllmascend,vllmmlu,sglang_ascend_npu}。这意味着每当上游发版、SDK 升级或 PyTorch 变更 API，三个环节都必须同步验证——任何一个掉链子，整体就可能退回到不可用状态。更麻烦的是，不同硬件上的 feature parity 不能默认成立：graph mode、attention backend、量化路径、PD 分离和 speculative decoding 等能力，即使在同一工具链中也可能存在支持粒度差异\citep{fastdeploy_docs,lmdeploy_supported_models}。这就解释了为什么社区经常看到"某个功能在 A 卡上能跑，在 B 卡上就不行"的报告——不是 bug，而是能力边界还没收敛成稳定契约。

\subsection{状态治理主路径：复杂任务场景的能力补齐}

如果说执行路径的问题是"跑不跑得动"，那么状态治理路径的问题就是"记不记得住、调不调得好"。当前真实负载已远不止单轮文本生成——长上下文、多模态、结构化输出、工具调用和多 Agent 协同\citep{dong2024xgrammar,liu2025elasticmm,xie2025aimetropolis}，每一种都在改变"状态"这个词的含义。

文本 serving 时代，系统管理的状态对象相对简单：请求队列、KV Cache 和 batch 形状。进入多模态和工具调用场景后，运行时还需管理视觉前缀、语法约束状态、工具回注上下文、外部执行等待与流式中断恢复——它们既非纯粹的模型内部变量，也无法完全交给服务层处理，而是直接影响调度顺序、缓存复用和回退路径选择的运行时实体。结构化输出要求把约束执行压进采样环路，工具调用要求支持生成与外部执行之间的往返切换，多模态请求则要求消化更长的前处理链和更剧烈的 shape 波动。对国产平台，难点不在接口名义上"可用"，而在这些能力会同时冲击调度、缓存和图模式三个层次。

推测解码进一步抬高了要求。SpecInfer、Medusa、EAGLE 与 Lookahead Decoding 分别代表辅助模型、多头预测、特征不确定性建模与并行前瞻路线\citep{specinfer2023,leviathan2023fast,cai2024medusa,li2024eagle,fu2024lookahead}。但收益高度条件化：在 MI300X 上的实验表明，speculative decoding 主要受益于小 batch 的带宽受限 decode，batch 增大后收益明显收缩\citep{singh2025specdecmi300x}。对国产平台，一个"只是多一次验证"的机制，迁移过来后可能同时意味着图缓存失效和异常回退链变长——系统复杂度的增长远超表面看到的那一步。

\subsection{压缩与成本主路径：收益不稳定与闭环缺失}

压缩路径的麻烦在于，它并不是一个可以独立解决的问题——它和执行路径、状态治理直接纠缠在一起。执行路径上的平台异构会限制低比特算子的可用边界，状态治理层的缓存组织则决定压缩策略能否在运行时稳定兑现。三者缺一，压缩收益就只能停留在离线实验阶段。

"低成本"听上去像是部署最后一步才需要关心的经济性指标，其实不然——它与执行效率和状态治理同级。问题在于压缩策略的收益往往不线性兑现：同一组量化参数在不同后端、不同上下文长度和不同负载强度下，可能分别表现为吞吐提升、尾时延恶化或精度波动。QServe 说明只有把量化、kernel 组织和 serving runtime 一起设计，低比特收益才可能在真实服务路径中落地\citep{wang2024qserve}；KIVI 与 SnapKV 表明长上下文场景下的状态压缩如果不和恢复路径与质量门限一起考虑，容量节省就会转化为恢复时延或命中率损失\citep{liu2024kivi,li2024snapkv}。在国产平台上这种耦合还会被放大——许多后端对低比特算子、动态 shape 和异步搬运的支持边界互不一致，让原本在 NVIDIA GPU 上验证过的压缩策略未必能直接迁移。

\subsection{统一底座：上游演进与评测体系}

如果说前面三条路径上的挑战各有侧重，那么"统一底座"层面的问题则是它们共同的放大器。底座不只是技术架构中的抽象层，更指系统在持续演进中能否维持工程秩序——这取决于两件事：与上游的关系怎么处理，以及用什么来度量系统的真实状态。

先说上游演进。高性能推理框架仍在快速变化，国产后端如果通过侵入式修改共享路径获得短期收益，后续将面临高昂合并成本。本地分叉之所以危险，还有一层更隐蔽的原因——它会逐步改变团队的优化激励。若长期依赖"复制上游模块 + 本地修补"的模式，团队最容易积累的是"快速打补丁"的经验，而非"设计稳定扩展面"的能力。反过来，若平台特化更多收敛在插件、注册表和受控配置门面中，团队就更有机会把每次适配劳动沉淀为可复用的工程机制。所以 merge-safe 不只是"对上游友好"的姿态，而是国产推理引擎能否持续演进的核心条件。

再说评测。当前不少优化仍围绕静态 benchmark 展开，但真实业务包含请求长度分布变化大、多租户并发、冷热模型混部和 SLA 约束严格等特征。如果评测只覆盖理想化场景，那么团队优化的方向和生产环境的真实需求就会持续脱节。未来评测需要从"单点性能测试"扩展为"端到端工作负载评测"，将 TTFT、稳定吞吐、缓存命中率、异常恢复时间和资源成本纳入统一框架。OpenCompass 与 FlagPerf 分别代表了模型能力评测和系统性能评测的生态努力\citep{opencompass,flagperf}。真正有价值的评测应当把这两类视角连起来——既看模型效果，也看支撑这些效果付出了多少系统成本。评测体系需要分层：底层算子用 microbenchmark 验证，运行时调度用受控 serving workload 验证（重点是 TTFT、TPOT 与尾时延），平台演进能力和运维代价则更适合通过持续集成基线、版本升级回归和故障演练来评估。
