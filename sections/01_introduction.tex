\section{引言}

大模型应用的快速扩张正在推动推理系统从单机实验环境转向面向生产的高并发服务平台。随着国产 AI 加速芯片、服务器整机与集群软件生态逐步成熟，围绕国产算力构建高性能、可维护、可扩展的推理引擎，已成为学术界与产业界共同关注的方向。但“国产算力推理引擎”并不是把已有 serving 框架迁移到另一类设备上的简单重复问题。相比通用硬件环境，本土平台往往同时叠加了设备能力离散、编译链约束差异、运行时环境治理复杂和交付路径多样等因素，使很多在主流生态中可以被局部吸收的问题，在国产场景下会迅速上升为系统级问题。也就是说，国产推理引擎的研究价值不只在于呈现一组本地适配结果，而在于揭示当平台边界、运行时状态和工程闭环一起变化时，推理系统的设计逻辑会发生怎样的重心迁移。与训练系统相比，推理引擎更加关注请求到达过程中的动态批处理、显存占用与复用、流式输出时延、长上下文处理能力，以及在工具调用、多模态输入和结构化输出等复杂任务上的稳定表现。对于国产算力平台而言，除了共享这些通用问题，还需要额外解决硬件后端差异、算子完备性、编译与运行时兼容性、以及上游框架快速演进带来的维护压力。

近年来，以 vLLM 为代表的推理框架通过 PagedAttention、连续批处理与高效内存管理显著改变了大模型服务系统的设计范式\citep{kwon2023efficient,vllm}。与此同时，SGLang 等系统进一步强调请求编排、结构化控制流与复杂推理工作负载上的执行效率\citep{sglang}。这些上游路线在 CUDA 生态上演化较快，但迁移到国产算力环境后，系统设计会从“单一硬件后端优化”转变为“统一框架抽象与本土设备能力之间的持续协调”。因此，国产推理引擎问题并不只是做一个新后端，而是需要重新评估引擎的抽象边界、能力优先级与可维护性策略。如果把视野放宽到近年的通用 serving 文献，可以看到推理系统的研究重心也在持续外移。Orca 与 Sarathi-Serve 说明，迭代级调度、连续批处理与 chunked prefill 已经成为高吞吐服务的基础组织方式\citep{orca2023,sarathi2023}；DistServe 与 Splitwise 则进一步表明，prefill 和 decode 的阶段解耦正在从局部优化上升为整体 goodput 设计问题\citep{zhong2024distserve,splitwise2024}；再到 XGrammar 与 ElasticMM，这种演进已经延伸到结构化执行与多模态并行，说明“推理引擎”正在从 token 生成器演变为状态密集型运行时\citep{dong2024xgrammar,liu2025elasticmm}。对国产平台而言，这些工作的价值并不在于简单移植某一项技巧，而在于它们揭示了哪些抽象边界最容易在复杂负载和异构后端上先失效。也正因此，本文虽然会引用 Llumnix、SOLA、LServe、XGrammar、ElasticMM 等近年的代表性系统工作作为参照，但这些文献在本文中的作用是帮助识别调度、缓存、多模态和结构化执行的前沿方向，而不是把讨论重心转移到脱离国产平台约束的通用 serving 综述。本文最终关心的判断标准，仍然是这些机制在国产芯片、国产编译运行时和本土部署环境中是否可迁移、可维护、可稳定落地。

从产业需求看，国产算力推理引擎面向的已不只是学术基准测试，而是包含在线问答、代码助手、政企知识服务、具身智能中间层和 Agent 工作流等多种 AGI4S 场景。这些场景对系统提出了更细颗粒度的要求：一方面要求较高的吞吐密度和资源利用率，另一方面又要求流式首 token 时延、上下文切换开销、错误恢复能力和服务可观测性达到生产标准。换言之，国产推理引擎的竞争焦点正在从“能否完成基础部署”转向“能否在真实业务下稳定、高效且可持续演进地运行”。这也解释了为什么本文需要把“技术路线”与“组织方式”放在同一分析框架中讨论。开源主干后端适配、平台一体化方案和混合式架构看似都在解决国产部署问题，但三者真正分开的不只是优化手段，而是对系统边界和演进节奏的不同假设：前者假定上游复用价值最高，中者假定平台统一性交付优先，后者则试图在接口兼容与本土特化之间建立较稳定的折中。只有把这些假设说清楚，综述才不至于退化为“列项目、列功能、列性能”的静态汇编。也正是在这一意义上，国产算力推理引擎的综述不能只停留在“支持了哪些模型”或“峰值吞吐达到多少”的罗列上。相比训练侧更容易用 FLOPS、集群规模等指标描述阶段进展，推理侧的核心矛盾往往分布在请求形态、缓存复用、阶段干扰、故障恢复与升级节奏这些不易被单一数字概括的维度上。尤其当模型族从纯文本扩展到多模态、语音和 Agent 工作流后，系统是否能够在较长时间尺度内维持接口兼容、性能稳定和运维可控，往往比某一次局部 benchmark 的领先更能决定其工程价值。

基于上述背景，本文不试图枚举所有通用 LLM serving 工作，而是引入一个更贴近当前国内推理引擎建设任务的分析骨架。其底座是统一抽象、统一指标和可观测支撑，用来回答系统如何在多平台、多版本和多阶段优化条件下保持可组合、可比较和可复现；其上是三条直接决定性能与交付的主路径：执行与通信主路径，关注执行图、图模式、算子与跨节点协同如何收敛到稳定高效的执行骨架；状态治理主路径，关注前缀复用、KV Cache 分层驻留、状态迁移与调度协同如何支撑长上下文和高并发；压缩与成本协同主路径，关注混合精度、KV 量化、稀疏--量化协同以及单位 token 成本如何进入系统级优化闭环。后文对系统架构、典型路线、关键挑战与趋势判断的讨论，都围绕这一“一个底座、三条主路径”的框架展开，并把“面向国产算力的可演进性”作为全文的主评价尺度。具体而言，第2节从核心架构层面解释这一骨架为何成立，第3节据此比较不同技术路线把复杂度压在何处，第4节讨论三条主路径在国产环境下如何相互放大挑战，第5节则据此推演未来演进方向。按照这一思路，本文的目标不是给出某一条路线的单向背书，而是建立一组更适合国产场景的判断准则：哪些能力应当被沉淀为平台契约，哪些状态必须进入运行时统一管理，哪些优化值得以插件或外围工具链的形式独立演进，以及哪些 benchmark 才足以支撑路线选择。若这组判断准则能够成立，那么国产推理引擎的比较就不再依赖零散经验，而更接近一套可复用的系统分析方法。
