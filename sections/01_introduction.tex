\section{引言}

大模型推理系统正从单机实验环境走向面向生产的高并发服务平台。随着国产 AI 加速芯片与集群软件生态逐步成熟，围绕国产算力构建高性能、可维护的推理引擎，已成为学界与产业界共同关注的方向。然而，"国产算力推理引擎"并不是把既有 serving 框架迁移到另一类设备上的简单重复：国产平台往往同时叠加设备能力离散、编译链约束差异、运行时治理复杂和交付路径多样等因素，使许多原本可以局部吸收的性能或工程问题迅速上升为系统级问题。

以 vLLM 为代表的推理框架已通过 PagedAttention、连续批处理与高效内存管理改写了大模型服务系统的设计范式\citep{kwon2023efficient,vllm}；SGLang 进一步强调请求编排与结构化控制流\citep{sglang,sglangpaper}；Orca 与 Sarathi-Serve 确立了迭代级调度与 chunked prefill 作为高吞吐服务的基础组织方式\citep{orca2023,sarathi2023}；DistServe 与 Splitwise 将阶段解耦从局部优化上升为整体 goodput 设计问题\citep{zhong2024distserve,splitwise2024}；XGrammar 与 ElasticMM 则把推理引擎推向结构化执行与多模态并行\citep{dong2024xgrammar,liu2025elasticmm}。当这些系统进入国产芯片和本土部署环境后，关键挑战会从硬件后端优化转向统一框架抽象与本土设备能力的持续对齐。

从公开生态观察，开源主干复用增强路线、平台一体化方案与国产原生/独立路线三类技术路径已经形成分化。三者的根本差异不只是优化手段不同，更在于对系统边界和演进节奏的假设不同。本文不把综述写成项目名录，而是围绕"生态格局与路线分化—分析骨架与核心架构—基于骨架的路线比较—结构性挑战—未来收敛方向"这一主线，采用"一个底座、三条主路径"的分析骨架展开讨论：底座是统一抽象、统一指标和可观测证据链；三条主路径分别是执行与通信、状态治理、压缩与成本协同。通过这一骨架，本文比较不同路线究竟把复杂度主要压在 backend、platform runtime、service state machine 还是 toolchain/control plane，进而识别哪些能力应沉淀为契约、哪些状态必须纳入运行时统一管理、哪些优化适合以插件形式独立演进。

本文的讨论建立在可交叉核验的公开材料之上。证据按四层使用：系统论文与技术报告界定机制脉络；官方文档与开源仓库确认能力边界；标准或白皮书辅助刻画生态趋势；社区部署记录仅补充真实摩擦点。对仅见于单次演示或未公开复现链路的材料，不据此给出性能与成熟度结论。路线比较围绕可演进性、状态治理能力、复杂负载承载能力和交付确定性展开，不试图在缺少统一复现实验条件的前提下给出跨平台绝对性能排序。
