\section{未来演进方向}

下一阶段，国产算力平台上的大模型推理竞争将继续从局部内核优化转向系统能力收敛。图\ref{fig:outlook-map} 所概括的几条收敛方向共同决定了系统上限：平台边界、运行时组织、统一评测与交付治理能否同时稳定，将比单个热点算子的局部提速更能拉开差距。未来的差距，主要体现在谁能把异构硬件、复杂负载和快速演进的模型生态压回同一套可持续的工程框架，而不是把问题分散为若干彼此割裂的性能补丁。

未来演进大致受三组相互牵动的收敛压力支配：一是平台抽象与运行时能否围绕能力契约和状态中心化共同演进，二是压缩、成本、评测与交付能否进入统一闭环，三是模型结构变化会如何继续反向塑造系统边界。这些变化并非彼此独立的新功能，而是同一套系统底座在未来几年要同时承受的压力测试。

\subsection{平台抽象与状态中心化运行时}

未来首先拉开差距的，往往不是单点性能数字，而是平台抽象能否从“后端接入机制”演进为“能力契约”，以及运行时是否真正围绕状态中心组织。随着上游框架持续扩展多模态、结构化输出和 Agent 场景，国产后端若仍依赖侵入式 patch 和深度分叉，就难以长期吸收社区演进成果。更可持续的做法，是把平台差异收敛为可合并、可替换、可验证的模块，并将 attention backend、图模式、动态 shape、通信语义、低比特路径和前处理约束显式化，使调度器、模型注册器和服务层能够基于能力而非经验特判做决策。对本土项目而言，平台插件化若能进一步沉淀为稳定契约，其价值会明显高于单轮适配带来的局部性能收益。

与此相伴，围绕长上下文、多模态与复杂控制流的资源管理将继续成为运行时优化中心。Prefix Cache、分层 KV Cache、跨实例迁移、SLO 感知调度和多模态阶段解耦，本质上都在处理同一个问题：如何在不确定负载下稳定组织状态。Llumnix、SOLA、LServe、AI Metropolis 与 ElasticMM 等工作已经说明，后续优化重点会更多落在状态迁移、依赖感知执行、统一稀疏注意力和多模态弹性并行等机制上\citep{sun2024llumnix,hong2025sola,yang2025lserve,xie2025aimetropolis,liu2025elasticmm}。对国产平台而言，这一点尤为关键，因为图编译成本、shape 敏感性和设备差异往往会把局部调度问题直接放大为端到端服务问题。更稳健的方向，不是继续把新能力外挂到 token 循环之外，而是把请求状态、缓存状态、约束状态与回退状态纳入统一状态机。

“平台抽象升级为能力契约”最终要落实到更细的运行时接口上：系统需要显式表达某类后端支持哪些 attention 变体、哪些 shape 区间适合图模式、哪些量化路径具备稳定 kernel、哪些通信拓扑允许 PD/EPD 扩展，以及异常发生时允许怎样的降级与回退。只有这些条件化能力被稳定表达出来，调度器和服务层才能在请求进入时做基于证据的选择，而不是依赖经验性配置模板。对国产推理生态来说，这一步比继续堆叠零散 patch 更关键，因为它决定后续每一次新模型接入和新平台纳入，究竟是在扩展能力矩阵，还是在继续扩大共享路径中的隐式假设。

\subsection{压缩、成本与交付控制面闭环}

压缩策略与成本核算也会从离线配置问题转向在线协同问题。未来若要真正降低单位 token 成本，系统不能只记录某组量化参数在离线评测中的精度损失，还必须同时知道它会怎样改变图模式命中、状态驻留、迁移带宽与尾时延。谁能更早把混合精度、KV 压缩、稀疏化路径和运行时观测组织成统一收益--代价闭环，谁才更有可能把“低成本”沉淀为稳定能力，而不是停留在阶段性实验结果上。

但成本闭环若无法接到评测与交付控制面，最终仍然难以转化为真实生产力。不同后端只有在统一接口和近真实负载下被比较，优化方向才不会被展示型 benchmark 带偏；部署环境中的驱动、编译链、插件版本和依赖树只有被纳入自动化诊断、回归和版本矩阵校验，系统上线与回滚成本才不会持续上升。作为第二类路线中更偏向主链复用与外围治理拆分的开源主干复用增强实践，vLLM-HUST 将平台插件、运行时护栏和外部环境治理拆分组织的做法，已经体现出“引擎主体 + 治理配套工具链”的演进方向\citep{vllmhust}。这也与 2025--2026 年公开社区高频出现的版本对齐、PD 分离、CANN/torch\_npu 兼容和 RoCE/HCCL 连通性问题相一致\citep{zhihu_vllm_ascend_guide,zhihu_vllm_pd_migration}。

在更具体的工程形态上，未来国产推理系统很可能会把“控制面”作为与执行面并列的一等对象来建设。控制面并不只是部署脚本集合，而是负责维护版本矩阵、平台能力画像、压缩策略基线、图缓存策略、回归用例、故障注入记录和灰度发布规则的统一治理层。执行面负责生成 token，控制面则保证这些 token 处于可复现、可回退、可比较的条件之下。只有当控制面本身被产品化，单位 token 成本、性能回归和环境稳定性才可能在同一条流水线上被持续经营；否则，许多优化成果仍会停留在开发人员的局部经验中，难以沉淀为组织能力。

\subsection{模型结构变化持续反向塑造系统边界}

从更长周期看，国产算力推理生态正在从“一芯一后端适配”走向多种抽象并行推进：vLLM-Ascend、vLLM-MLU 与 SGLang Ascend 体现了主流 serving 框架对国产后端的插件化或主线吸纳，FastDeploy、LMDeploy 把多硬件适配进一步工具链化，MindIE 代表平台一体化交付，Chitu 与 xLLM 则把状态治理、缓存与部署闭环内生到独立引擎；中国电信 Triton 路线把问题推进到统一算子、透明插件和跨芯图编译层，但其未开源和复现边界也说明，未来评价这类方案时需要同时考察技术抽象、公开版本矩阵与可验证交付链路\citep{vllmascend,vllmmlu,sglang_ascend_npu,fastdeploy_docs,lmdeploy_install,mindie,chitu_github,xllm_tech_report,chinatelecom_triton_validation_2025}。这里更值得关注的，不是再枚举一次路线名称，而是抽象层次本身正在重排，后端接入、运行时组织、状态治理与交付工具链的分界会继续移动。

模型侧的变化还会继续重写推理系统的受力点。以 DeepSeek 系列为代表的路线正在把问题从“模型结构更复杂”推进到“模型、推理内核、KV 基础设施与服务运行时是否一体收敛”：DeepSeek-V2/V3 披露的 MLA、MoE 与大规模服务经验，连同 FlashMLA、Mooncake 等配套工作，已经把通信组织、缓存布局、注意力实现和服务基础设施重新推回运行时中心位置\citep{liu2024deepseekv2,deepseekv3,deepseekflashmla,qin2024mooncake,miao2024mooncake}；DeepSeek-V4 则进一步把这种收敛扩展到更明确的软硬协同讨论，公开报告了相关内核在 HUAWEI Ascend NPU 上的验证，并提出面向硬件设计的 co-design 建议\citep{deepseekv4}。这表明大模型公司正在主动影响推理系统与硬件协同方向，而不再只是向既有引擎被动提出部署需求。与此同时，Qwen2-VL、InternVL2、CogVLM2、GLM-4-Voice 与 MiniCPM-o 等模型也说明，视觉、视频和语音链路已经使文本时代的单一路径假设难以维持\citep{wang2024qwen2vl,chen2024internvl2,wang2024cogvlm2,zeng2024glm4voice,cui2026minicpmo45}。在这种背景下，系统若仍沿用面向单模型、单平台、单 benchmark 的局部竞赛逻辑，就很难持续吸收模型结构和任务形态的变化。只有把能力契约、运行时状态机、评测闭环和交付治理一起推进，国产推理引擎才可能从阶段性适配进入可持续积累。

从运行时组织角度看，未来系统边界更可能围绕“阶段 DAG”而不是“模型家族”来重画。文本 decode、视觉 encode、语音前端、结构化约束验证、工具调用回注和多 Agent 依赖执行，本质上都是同一请求图中的不同节点；它们是否共用缓存、是否共享编译产物、是否需要独立回退路径，才是真正决定引擎形态的问题。若国产推理引擎仍按“先支持某个模型，再逐步补例外”的方式扩展，系统复杂度就会继续以模型名单的形式累积；而若能提前把阶段关系、状态迁移和控制平面显式化，模型变化就更可能被吸收为 DAG 上的新节点或新边，而不是共享主循环上的一串额外特判。这个转变未必立刻带来最亮眼的跑分，但更可能决定国产推理系统能否在未来几年内持续吸收新结构、新任务和新平台。
