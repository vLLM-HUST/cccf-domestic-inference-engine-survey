\section{未来研究展望}

前一节把路线比较收束为若干系统挑战，这些挑战如果长期得不到结构性回应，就会持续限制国产推理引擎的演进上限。那么未来真正值得投入的方向在哪里？我们的判断可以压缩为三条主线：面向异构平台的能力契约与共享抽象、面向真实交付的控制面与证据链治理，以及面向复杂请求图的状态中心化运行时。图\ref{fig:outlook-map} 概括了三条主线在国产推理生态中的收敛关系。

这三条主线并不是彼此独立的新功能列表，而是前文"一个底座、三条主路径"在未来更可能沉淀出的高层研究议题。它们之间存在递进关系：能力契约为系统演进提供稳定边界，控制面治理让这些边界变得可验证、可交付，而请求图重构则要求前两者同时升级以适应更复杂的运行时状态。

\subsection{面向异构平台的能力契约与共享抽象}

这条主线的核心问题是：平台差异能否从"各家项目内部的经验"上升为"生态可以共享的能力契约"。

目前的现状是，后端能力边界通常以隐式特判存在——代码里散落着各种 if-else，却缺少一套统一的描述语言来表达"这个后端能做什么、不能做什么"。要改变这一局面，首先需要回答的是能力应如何形式化表达：既要覆盖 graph/eager 约束、通信语义和低比特路径的真实差异，又不能把某个短期平台特例固化为长期接口。其次是回退语义的定义问题——当前系统对回退的处理通常是"降级到 eager"，但 graph mode 切换、缓存状态和分布式执行在回退前后的一致性，并没有系统性保证。第三个问题则关乎抽象的厚薄：若共享抽象过薄，平台差异反复溢出到应用层；若过厚，又把当前的短期特例固化为未来的包袱。Llumnix、SOLA、LServe 等工作之所以值得关注，正因为它们显示运行时组织方式正在脱离"单一模型 serving 外壳"，转向资源与状态中枢的形态\citep{sun2024llumnix,hong2025sola,yang2025lserve}。

只有当平台差异能够被收敛为共享能力矩阵——而非反复改写系统主链——新模型、新后端和新任务形态的进入才会从高风险迁移变成稳定的生态扩展。然而，能力契约本身并不能自动保证这些边界在生产环境中被持续遵守——这需要控制面来承担验证、回归和交付的闭环责任。

\subsection{控制面、评测与交付的一体化治理}

这条主线的核心问题是：控制面能否从运维脚本的集合演化为连接评测、成本与交付的一体化治理层。

从当前实践看，几个缺口最为明显。第一是近真实工作负载的标准化表达问题：目前缺少能同时覆盖请求分布、上下文长度变化、多模态混合和工具调用比例的统一 workload 描述格式。BurstGPT 等工作已开始提供真实负载数据集\citep{wang2025burstgpt}，但从数据集到可复现评测协议之间仍有显著距离。第二是版本基线与故障注入的统一问题——升级回归和灰度成本目前无法被量化治理，大多停留在人工经验阶段，让每次升级都像一场"盲飞"。第三是模型效果、系统成本与交付代价之间的证据链尚未连通，导致"性能结论"和"生产结论"长期脱节：论文里跑分好看的方案，到了生产环境可能因为运维成本过高而不可持续。

概括来说，执行面负责把 token 一个个生成出来，控制面则负责让 token 生成处于可比较、可复现、可灰度和可回滚的条件之下。只有把控制面视为与执行面并列的一等对象，前文讨论的成本、评测与交付问题才可能从局部经验转化为稳定的生态能力。而控制面治理的成熟度，反过来也取决于运行时本身能否提供更清晰的状态对象——这正是第三条主线关注的问题。

\subsection{面向多阶段请求图的运行时重构}

这条主线的核心问题是：当文本 decode、视觉 encode、语音前端、结构化约束验证、工具调用回注和多 Agent 协同被纳入同一服务链路后，推理引擎如何从维护单一 token 生成循环转向管理多阶段请求图。

这个转变并非遥远——DeepSeek-V2/V3 的 MLA 与 MoE 组织连同 FlashMLA、Mooncake 等配套工作，已经把通信组织、缓存布局和注意力实现重新推回运行时中心位置\citep{liu2024deepseekv2,deepseekv3,deepseekflashmla,qin2024mooncake}。这要求状态对象不再只按模型名单组织，而应围绕阶段关系来组织：哪些阶段可以并行、哪些阶段之间存在数据依赖、哪些中间产物值得缓存和共享。同样的逻辑也体现在模型侧——Qwen2-VL、InternVL2、GLM-4-Voice 与 MiniCPM-o 等多模态模型已经说明，视觉、视频和语音链路使文本时代的单一路径假设越来越难以维持\citep{wang2024qwen2vl,chen2024internvl2,zeng2024glm4voice,cui2026minicpmo45}。当新任务形态进入时，系统不应需要重写主链——理想状态是通过声明新的阶段关系和状态类型，让运行时自动完成调度和资源分配的适配。

这个转变未必立刻带来最亮眼的跑分，但更可能决定国产推理系统能否在未来几年内持续吸收新结构、新任务和新平台。系统边界会被模型公司、应用形态和硬件路线共同牵引，任何一方的变化都会迫使另外两方重写分工。

回过头来看，这三条主线共同指向同一个判断：国产推理引擎的下一阶段竞争，将从"谁更快适配某个硬件后端"转向"谁能在异构平台、复杂负载和快速演进之间建立更稳定的工程秩序"。能力契约、控制面治理和请求图重构不是三个孤立议题，而是同一问题的三个切面——系统能否把持续增加的复杂度组织进可扩展的边界，而不是让它们在外围不断堆积。
