\section{面向国产算力的大模型推理引擎生态}
\label{sec:domestic-ecosystem}

本章承接摘要提出的分析框架，从产业生态与技术路线落地的维度，拆解国产算力场景下推理系统的完整技术栈构成。综合公开生态与后文分析，国产推理引擎大致可归纳为\textbf{平台一体化方案、开源主干后端适配与国产原生/独立路线}三类：硬件厂商自研原生推理引擎对应平台一体化方案，围绕 vLLM/SGLang 等主流框架展开的硬件插件化与 backend 适配对应开源主干后端适配路线，而 Chitu、xLLM 一类不依附单一上游主干的独立引擎更接近国产原生/独立路线\cite{caict2024whitepaper,zhang2024domestic}。

面向国产算力的大模型推理引擎生态并不是单一软件系统，而是由硬件厂商基础软件、开源推理框架适配层、算子与编译栈，以及上层服务化平台共同构成。根据公开仓库与官方文档，可以将相关工作大致归纳为三类：第一类是硬件厂商或平台方围绕自有基础软件栈组织的原生推理引擎与部署体系，例如华为 Ascend MindIE、百度飞桨/昆仑芯生态中的 FastDeploy 原生部署栈、寒武纪 MagicMind、摩尔线程 MT-Transformer、天数智芯 IxRT/IGIE 等；第二类是围绕 vLLM、SGLang、LMDeploy 等主流大模型推理框架的国产硬件适配，其中硬件插件化正是这一类最典型的工程组织形态；第三类则以 Chitu、xLLM 等国产原生/独立引擎为代表。

三类生态实践对应了国产推理引擎研发的三条核心路径，其底层设计逻辑、技术取舍与目标场景存在本质差异，是后续进行技术选型对比的核心基础\cite{cui2025infer,sheng2024survey}。DeepSeek-V4 体现的是大模型公司推动的模型--系统协同演进，Ascend 则代表华为硬件与基础软件平台，两者分别对应模型侧与平台侧的不同演进力量\cite{deepseekv4}：
\begin{enumerate}[leftmargin=*, label=(\arabic*)]
    \item \textbf{硬件厂商自研原生推理引擎（平台一体化路线）}：核心逻辑为“全栈协同优化”。这一路线向下深度绑定自研硬件架构与基础软件栈，向上围绕 Transformer 算子、KV Cache、分布式通信与服务化接口做原生优化，把编译、运行时和交付模板尽量收敛在同一平台闭环中。MindIE、MagicMind，以及百度飞桨/昆仑芯围绕 PaddlePaddle 组织的原生部署体系，都更接近这一路线\cite{baidu2025fastdeploy,kunlun2024vllm_kunlun}。其优势是能够最大化释放硬件潜力，在 MoE 专家并行、低精度量化和内存层次优化等强硬件相关场景中更容易形成稳定收益；其代价则是生态兼容性相对受限，通常绑定特定硬件平台，迁移成本较高。
    \item \textbf{主流开源推理框架的国产硬件适配（开源主干适配路线）}：核心逻辑为“生态兼容优先”。这一路线以 vLLM、SGLang、LMDeploy 等主流开源主干为基础，通过后端适配、算子移植与插件扩展，把成熟的调度、缓存和服务接口迁移到国产硬件。硬件插件化是第二条路线最典型的技术组织方式：它通过 plugin backend、attention backend、distributed backend 等边界，把国产硬件能力压回上游框架的扩展接口中。vLLM-Ascend、vLLM-MLU、vLLM-Kunlun 与 SGLang Ascend 都属于这一类\cite{vllmascend,vllmmlu,kunlun2024vllm_kunlun,sglang_ascend_npu}。其优势是能够复用开源社区成果、保持模型与特性跟进速度；其约束则在于硬件特性释放受限于框架抽象层，并需要持续承受版本同步与插件维护成本。
    \item \textbf{国产原生/独立路线}：核心逻辑为“重组系统边界”。这一路线不再以给上游主干补后端为主要目标，而是围绕独立引擎、服务状态机、缓存治理和部署闭环重新组织推理系统。Chitu 这类独立生产级引擎属于这一类，xLLM 也更接近这一方向\cite{chitu_tsinghua_news,chitu_github,xllm_docs,xllm_service,xllm_tech_report}。其特点是更早把多阶段请求、多级 KV Cache、多硬件部署和运行时治理收进统一边界，因而更适合承载复杂工作负载；但系统边界设计难度更高，需要同时经营模型支持矩阵、部署入口和跨平台性能损耗。DeepSeek-V4 展示了与这一方向相邻的模型--系统协同演进，但并不是通用推理引擎产品\cite{deepseekv4}。
\end{enumerate}

从公开生态看，第二类路线中的 vLLM 硬件插件化正在成为国产算力适配的主要形态之一。vLLM-Ascend、vLLM-MLU、vLLM-MUSA、vLLM-MetaX、vLLM-Kunlun、vLLM-GCU 等项目均采用或参考 vLLM 的硬件插件机制，使厂商能够在尽量不侵入 vLLM 主干代码的情况下接入各自的 NPU、MLU、GPU、GCU 或 XPU 后端。这一路径的优势在于能够复用 vLLM 已有的连续批处理、KV Cache 管理、OpenAI 兼容服务接口、量化与投机解码等推理框架能力；挑战则在于底层算子、通信、图捕获、显存/片上内存管理、版本依赖和模型覆盖度仍需要各厂商持续适配。

vLLM 的硬件插件化架构经历了从\textbf{侵入式修改（In-Tree）}到\textbf{非侵入式插件（Out-of-Tree, OOT）}的关键演进。其核心变化是构建分层硬件抽象接口，在核心调度层、KV Cache 管理层和算子层分别定义标准化的硬件无关接口，使厂商通过实现对应后端即可完成接入，从而缓解早期适配中“一版本一修改、上游更新即失效”的维护问题。

当前，vLLM OOT 插件架构已成为国产算力适配的事实标准。国内主流硬件厂商大多基于该架构完成适配，行业内也在推进插件接口标准化，如中国信通院牵头的《大模型推理引擎硬件适配接口规范》以及上海人工智能实验室 DeepLink 生态定义的统一算子接口\cite{kwon2024vllm,caict2025standard,shailab2024deeplink}。但插件化架构仍存在以下核心痛点：
\begin{itemize}[leftmargin=*, label=\textbullet]
    \item \textbf{标准化程度不足}：不同厂商的插件实现仍存在碎片化，难以实现跨硬件的插件复用。
    \item \textbf{硬件特性释放不充分}：MoE 分布式通信、片上内存管理等强硬件相关特性，仍需侵入式修改主干代码才能实现最优性能。
    \item \textbf{维护成本依然较高}：国产硬件适配代码大多维护在厂商独立仓库中，难以进入上游主干，面临长期同步成本。
\end{itemize}

华为 Ascend 生态是目前公开资料较为完整的代表。一方面，MindIE 作为面向 Ascend 的 AI 推理加速套件，覆盖大模型推理、服务化和 PyTorch 生态适配等能力；另一方面，vLLM-Ascend、SGLang-Kernel-NPU 和 Triton-Ascend 分别从推理框架插件、SGLang 算子库和 Triton 编译后端三个层面补充了开源生态。这说明国产算力推理引擎的竞争点已经从单一模型执行扩展到框架调度、KV Cache、MoE 通信、算子融合、编译优化和服务化接口等完整技术栈。

除单一芯片生态外，跨硬件适配层也在快速发展。上海人工智能实验室 DeepLink 生态中的 DLinfer 试图在上层 LLM 推理框架与下层厂商融合算子或图引擎之间定义统一接口，LMDeploy 则通过 PyTorch Engine 和 DLinfer 等路径支持多类国产硬件。百度飞桨 FastDeploy 以 PaddlePaddle 为基础，面向文心等大模型提供推理部署能力，并在官方文档中列出对昆仑芯、Ascend、海光、天数智芯、沐曦、燧原等硬件的适配说明。与这些适配层和部署工具链并行演进的，还有 Xinference、GPUStack 一类服务平台，它们更侧重统一服务入口、模型编排和多硬件资源管理。这些项目的共同作用，是减少每个推理框架分别对接每个硬件后端的重复工程成本，并把部署与运维复杂度进一步从引擎本体外移。

该对比结果呈现了三类路线的主要技术取舍：不存在绝对最优的技术路线，只有与业务场景、硬件锁定程度和研发资源更匹配的选型。第三类也并非内部完全同质的成熟模板，而是对一组不依附单一上游主干、试图重组系统边界的对象所做的谨慎归纳，其中独立引擎与支撑平台并不简单同构。国产推理系统的核心竞争力，不在于局部算子的极致性能，而在于技术路线与场景需求的匹配度，以及能力契约、运行时组织和交付闭环的协同能力\cite{caict2024whitepaper,zhang2024domestic}。

上述生态显示出几个趋势。首先，国产算力适配正在从修改上游推理框架源码转向硬件插件化和后端解耦。这种方式能够降低维护成本，也便于跟随 vLLM、SGLang 等主流推理框架的快速演进。其次，厂商原生推理引擎仍然重要，尤其是在算子融合、图优化、MoE 通信、低精度量化和特定硬件内存层次优化方面，原生后端通常更容易发挥硬件特性；百度飞桨/昆仑芯围绕 PaddlePaddle 与 FastDeploy 的部署体系也更接近这一方向。再次，DLinfer、Triton-Ascend 等统一适配抽象层，以及 FastDeploy 这类部署工具链，说明第二类开源主干适配路线正在从单点插件开发走向“统一接口 + 厂商实现”的模式。

在生态快速演进的同时，国产算力推理引擎仍面临以下三大核心挑战，这些挑战也构成了本文后续引入“一个底座、三条主路径”分析框架的出发点：
\begin{enumerate}[leftmargin=*, label=(\arabic*)]
    \item \textbf{生态碎片化与标准化的矛盾}：国产算力生态目前仍存在“一硬件一栈、一框架一适配”的碎片化现象。不同硬件平台的基础软件栈、算子接口与通信语义差异巨大，导致行业重复适配成本极高。这要求系统必须在架构底层构建统一的抽象底座，以吸收平台差异\cite{cesi2024standard}。
    \item \textbf{开源生态兼容与自主可控的矛盾}：主流推理框架体系仍由海外社区主导，国产适配多处于“跟随式”状态。如何在深度兼容开源生态的同时，构建自主可控的执行架构与状态治理能力，是掌握技术演进话语权的关键。
    \item \textbf{性能极致优化与工程交付闭环的矛盾}：现有优化多聚焦于单算子或单模型的峰值性能，而相对忽视生产环境下的工程交付闭环。生产级推理系统的核心竞争力在于执行、状态与成本三者的协同优化，而非实验室环境下的指标罗列\cite{qian2024hetero}。
\end{enumerate}

上述矛盾表明，单纯的“算子替换”已不足以支撑国产推理引擎持续演进。第 3 节将引入“一个底座、三条主路径”的分析骨架，从系统架构层面展开讨论。
