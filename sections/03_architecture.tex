\section{分析骨架与核心架构}

当一个用户向推理服务发出一条请求时，引擎内部实际上同时在做三件事：执行模型计算（把 token 变成下一个 token）、管理状态（记住已经算过的东西，确定显存怎么分配）、协调服务（决定多个请求谁先谁后，结果如何送回去）。这三件事对应的就是本文"三条主路径"加"一个底座"的分析骨架。在成熟的 CUDA 平台上，这三件事彼此配合得很好，工程师通常不需要同时操心它们之间的关联。但在国产平台上，麻烦往往先出现在模块交界处——图模式回退时调度器知道不知道？多模态的前处理结果该存在哪里？平台插件到底只管设备发现，还是也得负责环境修复？表\ref{tab:key-tech-points} 把这个骨架下最核心的技术问题和评价维度做了一个概览。

\subsection{从最小推理闭环到统一分析底座}

最简单的推理过程可以想象成两步。第一步是"消化提示词"：系统把用户输入的整段文本并行计算一遍，建立起一个记录了"谁和谁相关"的缓存（KV Cache）——这就是 prefill。第二步是"逐个生成"：每次只算出一个新 token，然后更新缓存——这就是 decode。前者是计算密集型的（像一口气读完一篇文章），后者是带宽密集型的（像一个字一个字地写，每写一个都要回头看一眼前文）。这一差异是所有后续优化的起点：连续批处理、分块预填充、阶段解耦，本质上都在利用 prefill 和 decode 对资源的不同胃口\citep{orca2023,sarathi2023,zhong2024distserve}。

\subsection{执行与通信：决定系统能跑多快}

执行路径决定的是引擎能把模型算得多快、拓展得多广。要回答的核心问题包括：什么时候让编译器一次性优化整张图（图模式），什么时候回退到逐算子执行（eager 模式）；多张卡之间的通信拓扑怎么组织；以及如何在推测解码、分块预填充和 PD 分离中让计算和通信尽可能重叠起来。

对国产平台而言，执行路径还多了一层操心：维护成本。如果后端为了攒点性能在共享代码路径中塞了大量平台特例判断，一旦上游框架升级了模型执行接口或缓存布局，本地的补丁就会大面积失效。FlashAttention-3 直接利用 H100 的硬件异步执行和 FP8 能力，FlexAttention 则在保留融合算子性能的同时恢复了注意力变体的可编程性\citep{shah2024flashattention3,dong2025flexattention}。这类工作揭示的共同规律是：高性能后端的真正难点不在于"有没有 kernel"，而在于"如何把硬件的代际能力组织成可持续复用的接口"。

这就引出了国产平台执行路径的核心设计原则：把平台差异封装成可替换模块，而不是让它们渗透到共享代码中。更重要的是建立一套"后端能力契约"：明确告诉上层，哪些算子可以被图模式安全覆盖，哪些路径必须保留逐算子回退，哪些低精度路径需要调度器显式感知。国产平台的很多问题不是"后端不能用"，而是"某个能力只在特定条件下才能用"。如果系统没有办法表达这种带条件的能力边界，调度器就只能用越来越多的探测和分支来维持稳定——这正是当前许多国产后端的现状。

\subsection{状态管理：记忆怎么存、怎么复用、怎么回收}

推理引擎最显著的资源压力不在计算，而在记忆。每一次 decode 都要回头看前文的 KV Cache，而这个缓存随上下文增长会以线性速度占据显存。PagedAttention 的核心思想是把显存像操作系统管理磁盘一样分块管理，避免大块连续分配的浪费\citep{kwon2023efficient}。这一机制同时为前缀复用、分层缓存和跨设备迁移提供了基础。对国产硬件而言，状态系统还面临两个额外困难：不同芯片对内存分配粒度的偏好不同，而且当前缀复用、分块预填充和跨设备迁移同时开启时，缓存元数据的复杂度会急剧上升。

围绕注意力高效实现与缓存压缩已形成清晰技术主线。FlashAttention 系列从算子层逐步降低注意力计算的内存访问代价\citep{dao2022flashattention,dao2024flashattention2,shah2024flashattention3}；MInference、DuoAttention、vAttention 与 LServe 分别从动态稀疏注意力、检索头/流式头分离、虚拟内存管理和统一稀疏注意力执行改写了长上下文推理的系统组织\citep{jin2024minference,xiao2024duoattention,kwon2024vattention,yang2025lserve}；H2O、KIVI、SnapKV 与 CacheGen 则从重点击中、低比特量化、语义压缩和缓存流式传输角度缓解了 KV Cache 的显存与带宽压力\citep{zhang2023h2o,liu2024kivi,li2024snapkv,zhang2024cachegen}。

从更大的视角看，KV Cache 正在从执行层的附属模块演变为独立的基础设施。当长上下文、多轮会话和多 Agent 工作流同时存在时，缓存不仅要回答"数据放在哪"，还要回答"状态怎么计价、怎么回收、怎么复用、怎么迁移"。国产推理引擎在这方面更适合采用"统一抽象、平台特化实现"的方式：上层统一暴露块管理、引用计数、前缀复用和驱逐策略接口，具体的数据排布和搬运留给后端来实现。

\subsection{压缩与成本：不只是省显存的事}

很多人把量化和压缩看作部署完成后再"省一点显存"的后期技巧。但在国产平台上，压缩应当被视为与执行路径和状态管理并列的核心主路径。原因很简单：混合精度量化、KV 动态压缩和稀疏协同确实能降低显存占用，但它们能不能真正发挥作用，取决于后端有没有稳定的低精度算子路径、状态层能不能处理容量变化、调度器能不能吸收 batch 形态的波动。换句话说，压缩效果从来不由模型精度单独决定，而是由"压缩参数、执行效率、状态占用、服务质量"四个因素共同决定的。

如图\ref{fig:kv-cost-feedback-loop} 所示，压缩路径的工程价值需要回到 KV 组织、服务质量和单位 token 成本的联动关系中才能被正确评价。QServe 通过把权重量化、KV 压缩与 serving 系统协同组织，展示了低比特部署必须在 kernel、调度和缓存之间联合设计\citep{wang2024qserve}；KIVI、SnapKV 则进一步表明，长上下文场景下的 KV 压缩只有和缓存布局与热度判断结合，才可能不显著破坏服务质量\citep{liu2024kivi,li2024snapkv}。国产推理引擎若要把"低成本"转化为系统能力，就必须让压缩配置、运行时观测与成本核算使用统一接口。

\subsection{服务层收束：调度、接口与交付闭环}

\begingroup
\sloppy
服务层需要在吞吐与时延之间平衡。典型机制包括连续批处理、请求优先级控制、SLA 感知调度和流式返回。对国产硬件，还需考虑编译图缓存、静态形状约束与后端初始化成本——调度不仅要优化"请求何时执行"，还要优化"请求如何映射到平台最擅长的执行形态"。XGrammar 说明结构化生成已需要与推理引擎共同设计\citep{dong2024xgrammar}；AI Metropolis 进一步表明，多 Agent 负载下依赖感知的乱序执行是提高并行度的关键机制\citep{xie2025aimetropolis}。

在调度策略方面，DistServe 和 Splitwise 将 Prefill 与 Decode 的资源需求进一步拆解\citep{zhong2024distserve,splitwise2024}；Llumnix 通过跨实例动态迁移请求缓解负载失衡，SOLA 则把调度目标显式对齐到 TTFT 和 TPOT 等服务级目标\citep{sun2024llumnix,hong2025sola}。TensorRT-LLM 已把多模态支持做成相对完整的运行时能力集合\citep{tensorrtllmmultimodal2026}；OmniServe 尝试把低比特量化与长上下文统一稀疏注意力纳入同一 serving 框架\citep{omniserve2025}。这些工作虽不直接针对国产算力，但为理解调度协同、量化部署和工程化取舍提供了对照。

\endgroup

服务做到最后，落脚点是交付闭环。推理引擎面向外部的不只是一个 OpenAI 兼容接口，还包括监控、日志、链路追踪和故障恢复。这一层在国产系统中经常被低估，但在政企部署场景里，系统往往首先因为兼容性、可观测性或升级复杂度不过关而被淘汰。无论是 Transformers 的模型接口、Ray Serve 的服务编排，还是 Triton Inference Server 的部署体系，都在事实上定义了推理引擎需要兼容的工程边界\citep{transformers,rayserve,tritonserver}。

把以上三条主路径和底座放在一起看，国产推理引擎的架构难点并不在某个单一模块内部，而是集中在模块与模块的交界处：执行层和编译层的交界决定系统能不能跑，缓存和调度的交界决定长上下文和高并发下表现如何，对外接口和运维的交界决定系统能不能真正上生产。接下来讨论典型技术路线时，重点关注的就是不同系统在这些交界处做出了怎样的取舍。
